朝ラン計画ナビ（morning-run-navigator）

ランニング習慣化のための気象データ解析・スケジュール自動算定アプリ

1. プロジェクトの概要と解決した課題

私自身の「毎朝ベッドの中で天気予報アプリと雨雲レーダーを往復し、走るか休むかウダウダと悩む時間をなくしたい」という強い原体験から開発しました。心理学で言われる「決断疲れ（Decision Fatigue）」による朝一番の精神的エネルギーの浪費を防ぎ、ボタンをポチッと押すだけで論理的な最適解を即決してくれる相棒のようなアプリを目指しました。

2. 主要コンポーネント・使用技術

フロントエンドからバックエンド、インフラまで完全サーバーレスのモダンなWebストラクチャで構築しています。

  • フロントエンド: HTML5 / CSS3 / JavaScript (Vanilla JS) ※初期検証としてStreamlitを使用
  • バックエンド: Python 3.12 / FastAPI / Mangum (Lambdaラッパー)
  • インフラ・外部API: Amazon S3 / Amazon CloudFront (OAC経由) / AWS Lambda / Amazon API Gateway / Open-Meteo API (気象データ取得)

3. こだわりポイント・技術的アピール

  • ■ あえてデータベース（RDS）を作らない「エコ・インフラ」設計
    24時間起動で固定費がかかるRDS等のデータベースを一切構築せず、アプリに必要な都道府県マスタ（数KBのCSV）をバックエンドのZIPに同梱。Pythonの標準ライブラリでメモリ上に直接展開することで、運用固定費を「ほぼ0円（Route 53のドメイン維持費のみ）」に抑え込むコスト最適化を徹底しました。
  • ■ CloudFront OAC を用いた高セキュリティ静的ホスティング
    AWS SAAの設計思想に基づき、画面ファイルを格納する Amazon S3 のパブリックアクセスを全面ブロック。手前に Amazon CloudFront を配置し、OAC（Origin Access Control）によるアクセス制限を施すことで、S3への直接アクセスを遮断し、HTTPS暗号化通信に一本化したセキュアな王道インフラを構築しました。
  • ■ データ駆動型のシビアな減点アルゴリズム
    朝5時〜8時の降水確率・風速・体感温度データをシビアに評価し、傘が必要な雨なら大減点、向かい風が強すぎても減点といった、100点満点からの独自減点方式で採点。感情を挟まないデータ主導のスケジュール算定ロジックを実装しました。

4. 開発プロセスとトラブルシューティング

【検証からリファクタリング】 まずは最速でロジックを検証するため、PythonのみでUIまで作れるStreamlitでプロトタイプを開発。動作確認後、HTML/CSS/JSへのリファクタリングを行いました。その際、「関心の分離（Separation of Concerns）」を徹底し、インラインデザインを徹底的に排除したクリーンな外部CSS構成へ整理しました。画面間のデータ受け渡しには sessionStorage を採用し、軽量でクリーンな状態遷移を実現しています。

【環境依存エラーの撃破】 ローカルのMac環境からLinuxベースのAWS LambdaへFastAPIをデプロイする際、一部ライブラリのバイナリ互換性エラーに直面。これに対し、pipの --platform manylinux2014_x86_64 などのコンパイルオプションを名指しで指定してパッケージをビルドすることで、OS間の壁を乗り越えました。また、Lambda上でのカレントディレクトリのズレによるファイル読み込みエラーも、Pythonの sys.path.append を用いて絶対パスを強制追加するコードを仕込み、解決しました。

ドキュメント作成AIエージェント（DocuGen AI）

Gemini APIとビジネスフレームワークを融合した資料構成案自動生成アプリ

1. プロジェクトの概要と解決した課題

ブログの記事や仕事の設計書・提案書など、資料作成の最初の一歩で「白紙を前にして手が止まる時間」を無くすために開発した自分専用のアシスタントです。文章の「0から1の構成案」を考える作業に最もエネルギーを消費するという課題に着目し、最初の骨組みを爆速で組み上げる仕組みを作りました。

2. 主要コンポーネント・使用技術

バックエンドの機能とAIプロンプトの設計に開発リソースを集中させるため、軽量で高機能な構成を選定しています。

  • フロントエンド / バックエンド: Python (Streamlitフレームワークによる統合UI)
  • 生成AIエンジン: Gemini 2.5 Flash API
  • ファイル出力機能: python-docx (Word形式ダウンロード)

3. こだわりポイント・技術的アピール

  • ■ 他者公開を見据えた「APIキー課金・流出リスク対策」
    個人開発アプリをネット公開する際、最大の懸念となる開発者自身のAPIキー流出や予期せぬ巨額課金を防ぐ画面設計を導入。ローカル環境では環境変数（.env）から自動読込し、Web公開時にはユーザー自身に個人のAPIキーを入力してもらうサイドバー入力方式を採用し、安全に一般公開できる仕様に仕上げました。
  • ■ ビジネスフレームワークを組み込んだプロンプトエンジニアリング
    単に自由文章を作らせるのではなく、提案書にはロジカルシンキングの基本である「PREP法」、報告書には客観性を担保する「事実と解釈の分離」と「潜在リスクのアドバイス」、設計書にはシニアアーキテクトに扮した「代替案の技術提案」など、実務でそのまま叩き台として使えるプロンプトの構造化を行いました。
  • ■ 実務へシームレスに繋ぐワンクリックWord（.docx）出力機能
    ブラウザ上でのコピー＆ペーストの手間を省くため、生成された構成案をそのままビジネス文書として再利用できるよう、Wordファイルとして手元にダウンロードできる機能を実装し、利便性を高めました。

4. 開発プロセスとトラブルシューティング

【4時間での爆速個人開発】 面倒な画面のHTML/CSSコーディングを大幅に端折れるStreamlitを手段として柔軟に採用したことで、デザイン調整からネット公開までをわずか4時間という驚異的なスピードでやり遂げました。これにより、自分が最も頭を使いたい「本質的な課題解決（AIのプロンプトの練り込みや出力処理）」に時間を100%集中させることができました。

【状態管理の制約克服】 開発の途中、コードの修正・再レンダリングを繰り返す中でStreamlit特有の「duplicate ID（入力欄の識別子重複エラー）」の壁に直面。AIの生成コードを鵜呑みにせず、フレームワークが裏側で画面要素を一意に識別するための key 属性のライフサイクルと制約の仕組みを徹底して勉強し直すことで構造を理解。すべての入力フォームに適切な一意のキーを自力で再配分し、エラーを根本から解消しました。
